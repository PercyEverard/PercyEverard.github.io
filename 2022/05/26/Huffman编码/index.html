霍夫曼编码 (Huffman Coding) 是一种编码方法，霍夫曼编码是可变字长编码的一种。霍夫曼编码使用变长编码表对源符号（如文件中的一个字母）进行编码，其中变长编码表是通过一种评估来源符号出现机率的方法得到的，出现机率高的字母使用较短的编码，反之出现机率低的则使用较长的编码，这便使编码之后的字符串的平均长度、期望值降低，从而达到无损压缩数据的目的。霍夫曼编码是很多压缩算法的基础...

# 哈夫曼编码的优势

数据在进行传输的时候，信息都是表现为 0 和 1 这种二进制的形式。那么要想对数据进行传输，选择合适的编码方式非常重要。提到编码方式，我们可以联想起等长编码和变长编码。等长编码意思就是用同样长短的编码去表示所有信息。比如说我们这里使用 4 位二进制数来表示十进制数：

  • 0-->0000
  • 1-->0001
  • 2-->0010
  • 3-->0011

等长编码的优点很明显，每个字符都有对应的二进制编码，且长度是固定。所以很容易设计，也很方便读写。它的缺点是编码结果太长，很容易占用太多的计算机资源。我们常见的 ASCII 就是一种等长编码，将字符表示为 8 位特定的二进制数。

为了提高传输的速度，可以采用变长的编码方式，寻找更优的编码方式。同时，必须要保证编码不存在二义性（任意字符编码都不是其它字符编码的前缀。就比如说 A 是 00，B 是 0011，C 是 11，那么在解码的时候碰到 0011 是该把它解释成 AC 还是 B 呢？）。

哈夫曼编码是一种符合上述要求的编码方式，它采用自底向上的形式构造哈夫曼树。按照字符的概率分配码长，实现平均码长最短的编码。

# 哈夫曼树构建步骤

在构建哈夫曼树时要遵循一个原则，那就是权重越大的结点距离树根越近。因此，在构建 Halfman 树过程中，有如下的规律：

首先，选出我们数据中最小的两个数据，构建成二叉树的左孩子和右孩子，而根的数据为两者之和，并将根数据重新放进集合中。

其次，将刚才合成的数据作为右孩子，左孩子从未处理的数据中选出最小的一个，作为左孩子，他们的根同样为左右孩子的权值和。

不断重复上述的步骤，直到将所有的数据全部处理完并构建出二叉树，这棵二叉树就是我们的哈夫曼树！

# 哈夫曼编码分析

与等长编码相比，哈夫曼编码的长度减少，提高了传输速度。但在使用变长编码时，需要考虑一个字符的编码是否是另外一个字符编码的前缀，存在多义性是变长编码的缺点之一，这个时候如何合理地设计出变长编码显得尤为重要。

但是哈夫曼编码成功地解决了多义性这个问题。同时哈夫曼编码并非是一套固定的编码，它是根据给定的信息中每个字符出现的频率，动态地生成编码表。就假如说数据每进行一次传输之前都需要进行一次哈夫曼编码，那么两次传输虽然用到的字符都是相同的，但是由于字符出现的频率变得不一样了，所以这个时候编码表会动态地更新。

哈夫曼编码除了解决多义性，它还让编码的长度最小。这是哈夫曼树特性，它让树的带权路径长度的节点之和达到最小。与哈夫曼有关的名词详解可以参考这里。哈夫曼树的介绍及 C 语言代码实现 - 数据结构教程 - C 语言网 (dotcpp.com)

# 题外话

常见的几种编码方式 - 知乎 (zhihu.com)

我们常见的一些字符编码方式无非有 Unicode、ASCII、GBK、GB2312、UTF-8 之类的，如果在编辑时处理不当很容易出现乱码。我这里以顺序来描述上述编码方式的历史背景以及出现的契机，帮助大家更好地理解常见编码方式的优缺点。

ASCII 与 19 世纪 60 年代于美国诞生，用一个字节也就是 8 位表示一个字符，后面由于够用了就改成了 7 位表示一个字符，最高位设为 0。

对于英文国家而言基础版的 ASCII 已经够用了，但是欧洲其他国家的字符表示稍微比较麻烦点，128 个符号已经不够装了。所以即使在欧洲，不同的国家也都采用不同的 ASCII 编码格式。这种问题间接促成了 Unicode 编码的产生...

Unicode 编码包含了世界上所有的符号，但是它仍然是有问题的。比如说英文字母可以用一个字节表示，而汉字要两个字节才能表示出来，那么我两个字节的数据计算机是如何知道它是表示英文字母还是汉字呢？这就涉及到之前讲过的前缀问题。除此之外，就是 Unicode 用两个字节的二进制数来存储字符的话，那么英文字符前一个字节都是 0，岂不是浪费了存储空间？这就导致了 Unicode 字符集出现后仍然有各种的编码方式存在。

互联网的普及，强烈要求出现一种统一的编码方式。UTF-8 就是在互联网上使用最广的一种 unicode 的实现方式。它是一种变长编码格式。其他实现方式还包括 UTF-16 和 UTF-32，不过在互联网上基本不用。需要声明的一点是，这里的关系是，UTF-8 是 Unicode 的实现方式之一，Unicode 理解成一个符号集。

GBK/GB2312/GB18030，GBK 和 GB2312 都是针对简体字的编码，只是 GB2312 只支持六千多个汉字的编码，而 GBK 支持 1 万多个汉字编码。而 GB18030 是用于繁体字的编码。汉字存储时都使用两个字节来储存。
